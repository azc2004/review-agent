# [전역 개발 원칙 및 규칙]

## 1. 개발 환경 및 도구 (Environment & Tools)
- 기본 OS 환경은 macOS를 기준으로 작동한다.
- 패키지 및 소프트웨어 관리는 항상 Homebrew 또는 명시된 Python 가상환경 도구를 우선 사용한다.
- 에디터 환경은 VS Code 최적화 설정을 따르며, 불필요한 시스템 설정 변경은 금지한다.

## 2. 코드 스타일 및 아키텍처 (Code Style & Architecture)
- **언어 및 스타일**: Python을 주력으로 사용하며, 모든 함수와 클래스에는 PEP 8 표준, 명확한 타입 힌팅(Type Hinting), 그리고 Docstring을 필수로 작성한다.
- **아키텍처**: 결합도를 낮추기 위해 도메인 로직과 데이터 액세스 레이어를 철저히 분리하는 모듈러(Modular) 구조를 지향한다.
- **데이터베이스**: PostgreSQL, MariaDB 등 관계형 DB 접근 시 가급적 원시 쿼리(Raw Query) 대신 ORM을 활용하고, 조회 최적화를 위해 인덱스(PK 등)의 구조를 항상 먼저 확인한다.

## 3. 대용량 데이터 및 성능 최적화 (Data Processing)
- 수백만 건 이상의 대용량 상품 데이터나 벡터 데이터 마이그레이션 등을 처리하는 스크립트를 작성할 때는 메모리 누수 방지가 최우선이다.
- 대량 조회 시 한 번에 모든 데이터를 메모리에 올리지 말고, `yield`를 활용한 Generator 패턴이나 배치/청크(Chunk) 단위 분할 처리 로직을 반드시 구현한다.
- 예외 상황(Network Timeout, DB 연결 유실 등) 발생 시 데이터 유실을 막기 위해 멱등성(Idempotency)과 자동 재시도(Retry with exponential backoff) 로직을 포함한다.

## 4. AI 및 인프라 연동 규칙 (AI & Cloud Infrastructure)
- **LLM / API 연동**: 다양한 LLM 공급자(Claude, Gemini, GPT, AWS Nova 등)의 API나 Bedrock을 호출할 때는 비용 최적화를 위해 입력 토큰 청킹을 고려하고, 실패 시의 폴백(Fallback) 라우팅을 염두에 둔다.
- **클라우드 자원**: AWS, GCP 등 클라우드 인프라나 분산 검색엔진(Elasticsearch, Milvus 등)을 다룰 때는 비용 관리 가이드라인을 준수하며, 보안 그룹(Security Group)이나 접근 권한을 과도하게 개방하지 않는다.

## 5. 예외 처리 및 방어적 프로그래밍 (Safety & Error Handling)
- 모든 외부 API 호출 및 I/O 작업에는 타임아웃(Timeout)을 명시한다.
- 에러 발생 시 단순히 `pass`하거나 묻어두지 말고, 의미 있는 로깅(Logging)을 수행하며 기술 부채나 불완전한 구현이 발생한 곳에는 반드시 `# ponytail: [이유 및 업그레이드 경로]` 주석을 명시한다.

### [Rule: 과도한 의존성 및 오버엔지니어링 금지]
1. 단순한 문자열 파싱, 날짜 계산, 정규식 처리 등을 위해 새로운 외부 라이브러리를 설치하거나 제안하지 마라.
2. 코드를 작성하기 전, 반드시 다음 '의사결정 사다리'를 거쳐야 한다:
   - 1단계: 이 코드가 정말로 존재해야 하는가? (YAGNI 원칙 준수)
   - 2단계: 내장 표준 라이브러리(stdlib)로 해결 가능한가?
   - 3단계: 이미 프로젝트에 설치된 기존 의존성으로 구현 가능한가?
3. 위 단계를 거쳐 반드시 새로 작성해야 할 때만 '작동하는 최소한의 뼈대 코드'를 작성하라.
4. 만약 성능이나 마감 일정상 의도적으로 임시 숏컷(Shortcut) 코드를 작성했다면, 해당 코드 바로 위에 `# ponytail: [이유 및 향후 리팩토링 계획]` 주석을 명시하여 기술 부채를 기록하라.

### [Rule: 대용량 데이터 처리 및 I/O 최적화]
1. 수만 건 이상의 대용량 파일 데이터, DB 레코드, 혹은 벡터 데이터를 조회하거나 마이그레이션하는 스크립트를 작성할 때는 절대 전체 데이터를 한 번에 메모리(List 등)에 적재하지 마라.
2. 데이터를 청크(Chunk)나 배치(Batch) 단위로 나누어 처리하는 로직을 필수적으로 구현하고, 대량 조회 시에는 `yield`를 사용하는 Generator 패턴을 최우선으로 적용하라.
3. 데이터베이스(PostgreSQL, MariaDB 등) 연결 시 테이블의 인덱스(PK, FK 및 자주 조회되는 컬럼)를 먼저 파악하고, 불필요한 풀 스캔(Full Scan)이 발생하지 않도록 쿼리를 최적화하라.
4. 네트워크 타임아웃이나 DB 데드락 등 일시적 장애에 대응할 수 있도록, 지수 백오프(Exponential Backoff)를 포함한 자동 재시도(Retry) 메커니즘을 예외 처리 로직에 반드시 포함하라.

### [Rule: AI 모델 API 연동 및 비용 최적화]
1. 다양한 LLM 공급자(Claude, Gemini, GPT, AWS Bedrock 등)의 API를 연동하는 라우팅 코드를 작성할 때는 입력 토큰 크기를 사전에 계산하거나 제한하는 가드레일을 포함하라.
2. 주력 모델의 API 호출이 실패하거나 Rate Limit에 걸릴 경우를 대비하여, 자동으로 차순위 모델(가성비 모델 또는 타사 모델)로 요청을 전환하는 폴백(Fallback) 라우팅 로직을 반드시 설계하라.
3. 프롬프트가 과도하게 길어지는 것을 방지하기 위해, 컨텍스트 주입 시 불필요한 원시 로그나 중복 데이터를 필터링하는 전처리 단계를 구현하라.

### [Rule: 보안 및 접속 통제 (Security & Access Control)]
1. 데이터베이스, 외부 API, 클라우드 리소스에 접근하는 코드를 작성할 때는 반드시 접근 제어(IAM Role, Security Group)가 가장 느슨한 설정(Default) 상태로 배포되지 않도록 주의하라.
2. 관리자용 비밀번호(Admin Password), API 키, SSH Key 등 민감한 인증 정보는 코드 내에 하드코딩(Hard-coding)하지 말고, 반드시 환경 변수(Environment Variable)를 통해서만 로드하는 방식을 적용하라.
3. 개발 서버(Dev)와 운영 서버(Prod)가 동일한 설정 파일을 공유할 경우, 운영 환경의 민감 정보가 개발 환경으로 유출되지 않도록 설정을 분리하는 로직(Profile-based Configuration)을 구현하라.

### [Rule: Git Commit 메시지 작성 가이드]
1. 모든 Git Commit 메시지는 한국어로 작성해야 한다.
2. 커밋 메시지는 다음 세 부분으로 구성된다:
   - 제목: 최대 50자 이내로, 본문의 내용을 명확하고 간결하게 표현한다.
   - 본문: 변경 사항에 대한 자세한 설명을 2~3문장으로 작성한다.
   - 요약: 변경 사항을 한 문장으로 요약한다.
3. 커밋 제목 앞에는 다음 접두사를 붙여 커밋의 유형을 나타낸다:
   - feat: 새로운 기능 추가
   - fix: 버그 수정
   - docs: 문서 수정
   - style: 코드 스타일 수정 (포맷팅, 세미콜론 등)
   - refactor: 코드 리팩토링
   - test: 테스트 코드 추가 또는 수정
   - chore: 빌드 시스템, 패키지 매니저 등 설정 파일 수정
